# Ideation: Agent Runtime Next Optimizations

## Codebase Context

- The current wrapper layer already covers the original P0/P1 goals: worker contract, verifier pass, fixed `NOW` shape, intent-preserving compaction, cheap `explore` routing, docs, and regression tests.
- Real-task validation has been run, not just synthetic tests. The Jenny Wen feedback workflow task produced a valid worker report and a valid verifier report after retry.
- The biggest failures observed so far were not missing features. They were contract drift, preamble pollution, empty stdout, PowerShell encoding mismatch, and too-small verifier turn budget. Most of those have now been patched at the wrapper layer.
- The main residual fragility is that verifier success still depends on markdown contract compliance parsed by regex-like logic in [`scripts/runtime/orchestration-lib.mjs`](C:\Users\rickylu\.gemini\antigravity\scratch\financial-services-plugins\scripts\runtime\orchestration-lib.mjs).
- The main residual observability gap is that retries and failures are visible in terminal output, but not preserved in a durable machine-readable per-run ledger.
- The repo already contains mature patterns for `run-record`, `evaluation-summary`, and `scorecard` workflows under `financial-analysis/skills/autoresearch-code-fix/` and `financial-analysis/skills/autoresearch-info-index/`. Those patterns can be reused instead of invented from scratch.
- Current runtime tests are strong on entrypoints and local validation, but they are still mostly synthetic. The real task artifacts live under `runtime-state/`, which is gitignored and therefore not part of durable regression coverage.
- Current intent/session-state tests are English-only even though the operating language in this repo is frequently Chinese and mixed-language.
- Current operator workflow is documented, but still multi-step and manual.

## Ranked Ideas

### 1. Structured Verifier Mode
**Description:** Add a stricter machine-readable verifier output mode, ideally dual-channel: human-readable markdown plus a JSON or NDJSON sidecar generated by the wrapper contract. Keep markdown for operator readability, but validate and persist the machine form as the real gate.

**Rationale:** This is the highest-leverage next correctness improvement. The current verifier still passes through a fragile text-format bottleneck. The real residual risk is no longer "worker forgot a heading"; it is "verifier looked mostly right but drifted just enough to break parsing or silently reduce trust."

**Downsides:** Requires revisiting prompt contract, parser compatibility, fixtures, and operator docs. If done poorly, it can create two conflicting truth sources.

**Confidence:** 91%
**Complexity:** Medium
**Status:** Explored

### 2. Runtime Attempt Ledger And Scorecard
**Description:** Persist every worker/verifier run attempt into a durable run ledger with fields such as task id, profile, attempt number, retry reason, max-turn bump, prompt size estimate, elapsed time, output-path, validation result, and final verdict. Add a small rollup summary or scorecard view.

**Rationale:** The repo already has good run-record patterns elsewhere. Bringing that discipline into the runtime wrapper gives immediate observability, trend tracking, and easier debugging of flaky tasks. It also sets up later benchmark-driven improvement instead of anecdotal tuning.

**Downsides:** More local artifacts to manage. If the schema is too ambitious, this turns into telemetry theater instead of practical debugging support.

**Confidence:** 88%
**Complexity:** Medium
**Status:** Unexplored

### 3. Prompt Budget And Task-Shaping Guardrails
**Description:** Add a preflight budget layer that estimates prompt size, flags evidence-heavy tasks early, narrows plugin/context scope, and can automatically choose a safer verifier shape such as trimmed evidence, summarized worker context, or task splitting hints before the runtime call.

**Rationale:** Real failures clustered around long, evidence-heavy tasks. The current retry loop helps after failure, but there is still weak prevention before failure. This is likely the next biggest reliability gain after structured verifier output.

**Downsides:** Prompt-size estimation is approximate. If the guardrails are too aggressive, they can hide context the verifier actually needed.

**Confidence:** 84%
**Complexity:** Medium-High
**Status:** Unexplored

### 4. Canonical Real-Task Reliability Suite
**Description:** Convert a small set of realistic wrapper tasks into durable fixtures and regression scenarios. Start with the Jenny Wen feedback workflow and add 2-4 contrasting tasks such as a Chinese-heavy request, a longer evidence-pack request, and a lower-risk explore task.

**Rationale:** The current tests prove the wrapper works in principle. This would prove it keeps working on the kinds of tasks the repo actually cares about. It also protects against future regressions in contract handling, routing, and prompt assembly.

**Downsides:** Real-task fixtures can drift, be verbose, and require ongoing curation. The suite needs careful scoping to avoid brittle model-dependent assertions.

**Confidence:** 86%
**Complexity:** Medium
**Status:** Unexplored

### 5. Multilingual Intent And Session-State Hardening
**Description:** Add Chinese and mixed-language fixtures for `INTENT`, `COMPACTION`, and `NOW` generation, then tighten extraction logic where needed so hard constraints and non-goals are preserved reliably across Chinese, English, and mixed prompts.

**Rationale:** This repo is not English-only. Current tests and heuristics lean English, while actual user requests often do not. This is a correctness problem, not a polish problem, because dropped constraints in Chinese are equivalent to dropped constraints in English.

**Downsides:** The parser will still be heuristic-driven, so this improves robustness rather than making intent extraction fully deterministic.

**Confidence:** 83%
**Complexity:** Low-Medium
**Status:** Unexplored

### 6. One-Command Runtime Orchestrator
**Description:** Add a single entrypoint that turns `raw request -> INTENT -> NOW -> worker -> verifier -> saved artifacts` into one safe command with predictable output locations and summary reporting.

**Rationale:** Operator friction is still higher than it should be. A one-command flow would reduce manual mistakes, make the wrapper easier to reuse, and pair naturally with a future attempt ledger.

**Downsides:** This mainly improves ergonomics, not correctness. If built before the structured verifier and ledger, it risks smoothing over deeper issues instead of fixing them.

**Confidence:** 79%
**Complexity:** Low-Medium
**Status:** Unexplored

## Rejection Summary

| # | Idea | Reason Rejected |
|---|------|-----------------|
| 1 | Worker JSON contract right now | Overlaps with structured verifier mode, but the verifier is the stricter gate and the higher-value first migration target. |
| 2 | Auto-subagent orchestration for wrapper tasks | Premature scaling. Current bottleneck is reliability and observability, not parallelism. |
| 3 | Deep vendor runtime surgery | Violates the wrapper-first strategy that has worked so far and increases blast radius too early. |
| 4 | Full memory-file routing / Dream Memory follow-on | Valuable later, but current correctness and state-discipline layers are not mature enough yet. |
| 5 | Security monitor expansion | Important, but not the next highest bottleneck for the runtime wrapper path that was just hardened. |
| 6 | Pure model/prompt tuning pass | Treats symptoms, not the remaining structural weaknesses. |
| 7 | UI dashboard for runtime runs | Nice once a ledger exists, but weak as a first step without durable machine-readable run records. |
| 8 | Automatic raw-request to profile classification | Useful, but explicit profile entrypoints are good enough for now; the bigger problem is prompt shape and verifier robustness. |
| 9 | More plugin surface expansion | Expands capability before reliability. Wrong order. |
